finalize

Effective Java P24-27 第七条目：避免使用finalize

Effective Java中提及，finalize需要注意的点：

  • 1.不要理解成C++的析构函数

  • 2.finalize执行的时机是任意的，不能确保及时的回收（特殊场景如：文件用完就要关闭

  • 3.finalize所在的线程速度要比该应用的其他线程优先级低，所以回收速度没法保证；表明不要明确依赖该方法（作者举了一个他同事的图形GUI的例子

  • 4.finalize会带来严重的性能损耗（作者举了一个例子，性能相差430倍；从5.6ns到2400ns

    • 解决方案：提供一个自定义方法，来回收该类的资源；而不是使用finalize来回收（结合try-catch模式，如流的处理，显式调用close

finalize的优点：

  • 1.提供一个安全保证，在开发者忘记的情况下可以且在垃圾回收起作用时能够释放对应资源（如FileInputStream、FileOutputStream、Timer、Connection

如FileInputStream中，在finalize中实现资源显式调用close释放IO资源（一定程度上纠正开发者的错误，避免产生不可以预料的情况，如文件无法再次被访问

  • 释放native对象的资源

当该Java对象如果有native对象的话，那么垃圾回收器触发了Java的finalize方法的时候，也应该进行native对象的回收

  • 需要注意，子类不会自动调用父类的finalize的方法，需要开发者显式调用

最佳的调用方式：（保证子类异常，父类也可以正常回收

  • 为了防止上述一点子类忘记调用父类的finallize（防止外部子类的恶意实现，父类要确保自己的回收）

作者提供了一个使用finalizer guardian的思想，通过一个匿名内部类，重写他的finalize的方法来实现父类自己的回收方法

作者通过这种方式，当子类（继承自父类Foo）被回收的时候，就算子类不调用super.finalize，由于finalizerGuration会被回收，所以会调用finalizerGuration的finalize方法，让父类Foo的finalize的方法得以被调用

小结：

  • 1.finalize是作为回收关键资源的方法，请勿随意重写

  • 2.如果使用了finalize的方法，一定要注意调用父类的super.finalize方法

  • 3.如果父类要方法子类的粗心，那么可以考虑使用Finalizer Guration的方式实现

深入理解JVM P66页

对象的生存与死亡：

  • 真正宣告一个对象死亡需要经过两次标记的过程

  • 当发现对象不存在于GC ROOTS的引用链时，对象会被标记一次，并会执行一次finalize（如果重写的了话，不重写不执行）

  • 如果有必要执行，JVM会把对象放到一个F-Queue队列中，并且稍后放入一个由JVM创建的低优先级的Finalizer线程中去执行它

  • 需要注意，JVM会触发这个方法，但是不保证执行完的时间（因为线程优先级低，这个和上面有一点是讲到的

  • 如果一个对象在finalize方法中执行缓慢，可能会导致F-Queue其他对象永远处于等待，甚至导致整个内存回收的机制崩溃

作用：

  • 引升为Java对象的“自我救赎”，可以通过在第一次finalize的时候，将该对象的引用赋值个某个一定存在于GC ROOTS上的变量（如静态量；可以一定程度上逃离被回收的境地

可以发现，我们不重写hashCode方法，用equals比较两个对象是相等的，但是产生的hashCode却不一样。

根据约定hashCode如下特点：

  • 1.在应用执行过程中，只要equals比较所得到的信息没有被修改，那么多次调用hashCode方法始终返回相同的数

  • 2.如果两个对象equals对象比较是相等的，那么hashCode必须相等

  • 3.如果两个对象equals不相等，那么hashCode不一定不相等（也可能相等）

  • 4.hashCode如果不相等是能有效提高Map一类集合的性能，因为冲突少了

强调/注意：

对象的hashCode()返回的不是对象所在的物理内存地址。甚至也不一定是对象的逻辑地址，hashCode()相同的两个对象，不一定相等，换言之，不相等的两个对象，hashCode()返回的哈希码可能相同。

小知识：

上述hashCode()的重写中出现了result31，是因为result31 = (result<<5) - result。之所以选择31，是因为左移运算和减运算计算效率远大于乘法运算。当然，也可以选择其他数字。

wait notify

注意两点：

  • 1.wait和notify要在同步代码块中使用

If wait(), notify(), andnotifyAll() methods can be called by a ordinary method , the reading thread callswait() and the thread is being added to waiting queue . At just the same moment, the writing thread calls notify() to signal the condition changes. The reading thread misses the change and waits forever. Hence, they must be called inside a synchronized method or block which is mutually exclusive.

原因是举了一个例子，如果不加入同步代码块，读线程调用wait与此同时写线程调用notify，那么读线程就可能永远失去了被唤醒的机会。 于是乎，wait和notify要在加入同步代码块
